Five things we'd add to the Knowledge Graph

The knowledge graph that's been built is strong — and it's honest about its own open questions. These are five places we'd strengthen it, in plain language.

These are our notes on top of the existing design, not a description of what's built today. The console mockup stays faithful to what exists; this page is where our additions live, so the two don't get confused. Each item is something the master spec either flags as a gap or leaves thin.
1

Make regulations a real thing in the graph

So you can answer an examiner's favourite question.
The gap

Today the graph holds rules ("balance under $2,500 means a $5 fee") and disclosures, but the actual regulation behind them isn't its own object — it's only implied. The spec admits this openly.

An examiner asks: "show me every action governed by Regulation E, section 1005.11." Right now that's hard to answer, because nothing in the graph is "Reg E §1005.11" — it's scattered across rules and disclosures.
What we'd add

A first-class Regulation object with its real citation, and every rule, disclosure and policy points back to the exact clause it comes from. Then "what does this regulation touch, and what breaks if it changes?" is one hop away — for examiners and for risk teams.

2

Separate "an agent can read it" from "an agent can say it"

Two different permissions, often confused.
The gap

Today knowledge is tagged mainly by whether it's personal data. But in banking, "the agent may use this" and "the agent may repeat this to a member" are not the same thing.

Overdraft fee mechanics: an agent should use them to answer correctly, but shouldn't volunteer the internal waiver thresholds to a member. One tag can't capture both.
What we'd add

Four clear levels instead of one:

Publicsafe to say to a member Internaluse it, don't repeat it Guardedonly certain agents Restrictedgovernance only
3

Decide what happens when knowledge is missing or stale

A confident wrong answer is worse than "I don't know."
The gap

The spec asks for gap and conflict detection, but doesn't yet say what the agent should actually do when it hits one. That behaviour needs to be a rule, not a guess.

If the answer is missingsay "that's not in the knowledge base" — never invent it
If it's out of datecaveat low-stakes facts; refuse on money, eligibility, rules
If two sources disagreetrust the attested fact over a system pull over the website
If it's restricteddecline and point to a human
What we'd add

These four behaviours, written down once and applied everywhere — so every agent fails safely the same way.

4

Connect "what the bank knows" to "who the members are"

Two graphs, joined by one link.
The gap

The knowledge graph deliberately holds no member data — which is right. But member accounts live in a separate instance graph (the one we proved with the fraud-ring demo). How the two connect at answer-time isn't spelled out.

◧ Knowledge graph
products, rules, disclosures — what the institution knows
↕  one link: this account is an instance of that product
◈ Instance graph
real members & accounts — pulled live from core systems
What we'd add

Define that single bridge so an agent can step from a real member's account, to the product it's an instance of, to the rule that governs it. The member data never enters the knowledge graph — they just touch through one well-defined link.

5

Prove the documents were actually understood

Uploading a policy isn't the same as capturing it.
The gap

The whole "ready in days" promise rests on reading the FI's documents correctly. But a policy can be in the system and still have half its rules un-extracted. The spec calls this out as an unsolved risk.

A 12-page overdraft policy is uploaded. The graph shows "overdraft policy: present" — but did it actually pull out all nine of the rules buried in the fine print? Presence isn't completeness.
What we'd add

A way to measure extraction, not just confirm a file exists — flagging rule-bearing sentences that were never turned into rules, so a human reviews the real gaps instead of trusting a green checkmark.
