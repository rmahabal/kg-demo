Storage — decided

v1 is a logical graph over ordinary storage (typed-reference records). The instance graph is a real graph DB (the proven Neo4j fraud-ring POC). The knowledge graph adopts a graph engine only when a trigger fires: live multi-hop queries, graph visualisation as a feature, or cross-FI analytics. The model ports either way.

The bridge — our spec to write

The knowledge graph holds no member data; the instance graph holds the members. They meet at one edge — account → instance-of → product — so an agent steps from a real account to the product, to the rule that governs it. Neither document specifies this join yet; it's ours to define.

16 objects · 4 kinds · one universal schema · per-FI

One graph, segments are subgraphs

The 16-object core fully covers Consumer. SMB / Business / Commercial add composable modules — Parties & Ownership (KYB), Lending & Credit, Product subtypes. Customer generalises to Party = Person | Legal Entity. One schema per FI; turning on a module lights up the blocked agent families.

What we'd add (see explainer)

A first-class Regulation object (so "everything governed by Reg E §1005.11" is one hop), four-way sensitivity (read vs say), and written serving rules for missing / stale / conflicting knowledge. * Customer & Account stay empty in the knowledge graph by design.
